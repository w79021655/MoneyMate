# 任務接收與範圍控制

## 開始前

1. 用一句話確認預期成果，而非只重述操作步驟。
2. 檢查 repository 狀態、目標檔案、直接呼叫端與既有測試。
3. 依 `task-routing.md` 讀取必要規則並以繁體中文回報。
4. 判斷任務是回答、診斷、實作、重構、review、文件或交付作業。
5. 能從專案安全推斷的細節直接處理；會改變 public contract、資料 schema、架構方向或使用者體驗的關鍵選擇需先確認。

## 範圍原則

1. 採取最小且完整的變更，不進行無關重命名、搬檔、格式化或 modernization。
2. 使用者要求診斷時只定位與說明，不自動實作修正。
3. 使用者要求實作時，完成必要程式碼與相稱驗證，不停在純建議。
4. 發現相鄰問題時先記錄並回報，除非它直接阻擋目前任務，否則不納入 diff。
5. 工作樹已有使用者變更時保留並避開；若與必要修改重疊且無法安全合併，再向使用者確認。
6. 新增 dependency、修改 deployment target、schema migration 或大範圍架構遷移視為重大擴張，需先取得同意。

## 文件與溝通語言

所有進度、分析、決策、Markdown 與完成回報一律使用繁體中文；不可翻譯的技術內容依 `CLAUDE.md` 保留原文。
